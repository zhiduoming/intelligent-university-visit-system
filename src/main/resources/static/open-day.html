Project Story

这个项目的重点不是 CRUD 数量，而是数据模型能不能解释择校问题。

UniTour 的核心设计是高校、校区、POI、评价四层模型。高校负责宏观画像，校区负责差异，POI 负责具体体验，评价负责真实反馈。

01 / Domain

高校挂载评价

主评价属于高校，因为择校决策首先比较的是学校；校区作为可选维度，表达“具体体验发生在哪里”。

02 / Cache

Redis 详情缓存

高校详情包含校区列表和评分汇总，读多写少，适合使用 Redis String JSON 缓存，并在评价变化时失效。

03 / Review

幂等点赞与逻辑删除

点赞使用唯一键避免重复写入；评价和回复使用逻辑删除，保留业务痕迹并降低误删风险。

后续可以继续做什么。

这部分不是假装已经实现，而是把下一阶段路线讲清楚。

开放日活动

接入高校开放日、宣讲会、报名入口和时间筛选。

收藏与对比

把目标高校加入对比池，横向比较评分、校区和评论。

POI 地图

接入地图 SDK，把现在的 POI 卡片升级成校园地图。

内容审核

评价内容可加入敏感词、举报、审核状态和管理员处理流。
