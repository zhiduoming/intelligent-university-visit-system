# **智能高校攻略引擎(UniTour) 需求与架构设计文档 v2.0 (开发版)**

## **1\. 系统定位与开发原则**

* **定位：** 一款集“校园导览、研学攻略、考学辅助、校友社交”于一体的综合性后端系统。旨在解决目前市面上官方招生信息过于枯燥干瘪，而社交平台信息又过度碎片化、缺乏系统性地理指引的核心痛点。  
* **开发原则 (解耦核心)：** \* 采用 **核心+插件（Core \+ Plugins）** 的微内核架构思想。  
  * **核心模块（高校、POI）保持绝对轻量：** 这些是高频读取的基础只读数据，绝不硬编码任何容易变更的业务逻辑（如具体的评分数值计算、复杂的组队流转状态）。  
  * **扩展模块（日记、评价、社交）实施单向依赖：** 所有插件模块只能通过外键（如 uni\_id, poi\_id）单向关联核心模块。这样能确保未来在添加新玩法（比如增加“校园寻宝打卡”插件）时，底层的旧表和旧代码做到真正的“零修改”，严格符合开闭原则（OCP）。

## **2\. 系统模块划分 (解耦蓝图)**

为了保证你一个人作为全栈开发者不会在复杂的逻辑中迷失，系统分为四大独立模块。请严格按照以下层级自底向上开发：

### **🟢 \[主体核心\] Module 1: 基础资源模块 (Core Domain)**

*这是系统的基石与骨架。如果这部分挂了或者数据不准，整个上层建筑都将失去意义。*

* **1.1 校园百科子系统：** 维护高校基本面、多校区地理数据。由于很多历史悠久的高校经历了多次合并或扩建（如北邮的西土城老校区与沙河新校区，两者氛围与功能截然不同），必须从系统底层支持“一校多区”的空间概念。  
* **1.2 深度点位 (POI) 子系统：** 维护校园内部的具体坐标及图文数据，细化到“哪一栋教学楼”、“哪一个食堂的哪个档口”。  
* **1.3 身份认证子系统：** 处理用户注册、登录（基于无状态的 JWT 鉴权机制，便于未来分布式部署）、角色划分（高中生/在校生/系统管理员）。

### **🟡 \[次要功能\] Module 2: UGC 内容社区模块 (Community Plugin)**

*这是一个挂载模块，赋予了系统“生命力”。即便该模块由于高并发宕机，核心的查阅导览功能依然必须保持可用。*

* **2.1 研学日记：** 用户参观后的长图文记录，支持富文本排版，用于沉淀高质量的“民间攻略”。  
* **2.2 愿望墙：** 轻量级、高频次的文字交互，记录用户的打卡愿望（如“明年顶峰相见”），主要用于提升初高中生用户的留存率与情感共鸣。  
* **2.3 评价中台：** 高度抽象的统一评分系统。采用多态设计，既可评“整所学校”，也可针对“某个具体 POI”进行打分。

### **🔵 \[后续扩展\] Module 3: 报考与决策辅导模块 (Admission Plugin)**

*将系统从“导览工具”升维至“决策智库”。*

* **3.1 开放日看板：** 实时同步高校参观预约信息，帮助用户规避“跑空”的尴尬，提供极具时效性的官方预约入口指引。  
* **3.2 多维雷达图：** 后端通过定时任务（Scheduled Tasks）聚合历史评价数据，动态生成各高校在“美食/宿舍/学术压力”等多维度的评分分布，提供直观的报考参考。

### **🟣 \[终极扩展\] Module 4: 社交与组队模块 (Social Plugin)**

*产生用户裂变与高频互动的终极利器，也是面试时最能体现你应对“高并发场景”能力的模块。*

* **4.1 校友寻访：** 基于“毕业高中”标签的同好自动匹配，建立基于地缘和学缘的微型社交图谱。  
* **4.2 跨校组队：** 核心难点在于并发处理。需要严密处理游览车队的招募发车名额、控制超卖问题，以及严格的组队状态流转闭环。

## **3\. 主体核心功能详细说明 (MVP 阶段必须完成)**

### **3.1 用户认证与管理 (User Auth)**

* **注册/登录：** 支持账号密码登录（数据库层必须使用 BCrypt 等强哈希算法对密码进行加盐加密，严禁明文存储）。成功后颁发具有时效性的 JWT Token。  
* **用户信息完善：** 首次登录需完善用户画像（选择身份为高中生或大学生）。根据身份动态展示不同的必填项：高中生选填“目标高校”，在校大学生则必须绑定“当前就读高校”。  
* **权限控制：** 引入 Spring Security 框架构建安全上下文。配置严格的拦截器策略：除 /api/v1/auth/\*\* 和部分公开的高校查询接口外，其余接口均拦截未携带合法 JWT 的请求。

### **3.2 高校与多校区管理 (University & Campus)**

* **数据结构层级：** 严格遵循 全国 (National) \-\> 大学 (University) \-\> 校区 (Campus) 的树状包含关系。  
* **业务逻辑：** \* 查询大学详情时，后端必须级联（通过连表或批量查询）返回其下属的所有校区信息。  
  * （原型数据落地：必须包含北京邮电大学的【西土城校区】：侧重历史底蕴与核心实验室；以及【沙河校区】：侧重现代化住宿与宽阔的活动空间）。

### **3.3 点位导览 (POI System)**

* **点位分类 (Tagging)：** 采用双轨制标签。硬性类别（枚举值：学术、生活、文化、风景）用于固定的 Tab 栏切换；软性标签（如：学长强烈推荐、便宜大碗、期末爆满）作为 JSON 数组存储，用于全文检索和个性化过滤。  
* **业务逻辑：** \* 前端通过传入 campusId 动态拉取对应 POI 列表，后端需支持基于坐标的距离排序（初期可简化为按热度排序）。  
  * 每个 POI 应当包含 suggested\_duration（建议游玩时长）字段，为后期算法自动规划连续路线打下数据基础。

## **4\. 扩展功能详细说明 (严格解耦，分期开发)**

### **4.1 UGC 中台 (日记/愿望/评价)**

* **解耦设计细节：** \* 评价中台必须设计为“通用评论系统”。核心在于包含 target\_type (1代表评学校，2代表评POI) 和 target\_id 这两个联合寻址字段。这样一来，即使未来业务线拓展到需要评价“某个组队活动”或者“某篇热门日记”，你只需要在代码中增加一个 target\_type=3 或 4，底层数据库表结构无需做任何 Alter 操作。  
  * 愿望墙数据价值较低且并发写入可能较高，物理设计上只存 uni\_id 和简短内容，与核心系统的关联降到最低。

### **4.2 开放日与预约**

* **逻辑设计：** \* 设计为独立的事件表 open\_day\_event，通过 uni\_id 绑定所属高校。  
  * **时效性过滤机制：** 后端 SQL 查询时必须默认加上 WHERE end\_time \> NOW() 的时间拦截，确保失效的开放日活动不会误导用户。同时可引入事件状态枚举（UPCOMING 未开始, ONGOING 进行中）。

### **4.3 社交组队 (难点)**

* **状态机设计：** 组队生命周期必须被严格管控，状态流转需遵循单向或闭环规则：招募中 (Recruiting) \-\> 已满员 (Full) \-\> 进行中 (In\_Progress) \-\> 已结束 (Finished) \-\> 已取消 (Canceled)。  
* **安全与并发设计：** \* 限制单个用户的同时发车数量（如最多处于 1 个活跃组队中）。  
  * 在处理“加入队伍”接口时，面对余量扣减，考虑在 SQL 层引入基于 version 字段的**乐观锁**（Optimistic Locking）机制，或者使用 UPDATE team SET current\_members \= current\_members \+ 1 WHERE id \= ? AND current\_members \< max\_members 的原子更新策略，向面试官展示你对“惊群效应”和超卖问题的防范意识。

## **5\. 核心数据模型 (数据库表设计指南)**

**开发提示：** 以下为物理建表的参考规范。除了业务字段外，强烈建议为每一张核心表加上大厂标配的**通用审计字段**（create\_time, update\_time, is\_deleted），采用逻辑删除（Soft Delete）而非物理删除，以保证数据的可追溯性。

### **🟢 核心表 (Core Tables)**

**1\. user (用户表)**

* id (PK, BIGINT, 推荐使用 Snowflake 算法生成分布式主键)  
* username (VARCHAR, 唯一索引)  
* password (VARCHAR, 长度需能容纳 BCrypt 密文，建议 100\)  
* role (TINYINT: 0-普通用户, 1-在校生, 9-管理员)  
* high\_school (VARCHAR, 需建立普通索引以加速校友匹配查询)  
* current\_uni\_id (BIGINT, 关联大学表, 在校生专属外键)  
* *通用审计字段* (create\_time, update\_time, is\_deleted)

**2\. university (大学主表)**

* id (PK, BIGINT)  
* name (VARCHAR, 建立唯一索引 uk\_name)  
* logo\_url (VARCHAR, 存储 OSS 或 CDN 链接)  
* description (TEXT, 学校历史与底蕴长文)  
* level\_tags (JSON 或 VARCHAR，如 \["985", "211", "双一流"\])  
* *通用审计字段*

**3\. campus (校区表)**

* id (PK, BIGINT)  
* uni\_id (BIGINT, 建立普通索引，因为经常需要 WHERE uni\_id \= ?)  
* name (VARCHAR, 如 "西土城校区")  
* address (VARCHAR)  
* lat, lng (DECIMAL(10,6), 保证坐标经纬度的足够精度)  
* *通用审计字段*

**4\. poi (具体点位表)**

* id (PK, BIGINT)  
* campus\_id (BIGINT, 建立普通索引)  
* name (VARCHAR, 如 "学一食堂")  
* category (TINYINT, 1-文化 2-学术 3-生活 4-风景)  
* suggested\_duration (INT, 建议游览分钟数，用于路线规划)  
* image\_url (VARCHAR)  
* intro (TEXT)  
* *通用审计字段*

### **🟡 扩展表 (Plugin Tables \- 独立挂载)**

**5\. ugc\_wish (愿望墙表)**

* id (PK, BIGINT)  
* user\_id (BIGINT)  
* uni\_id (BIGINT, 许愿的目标学校)  
* content (VARCHAR(100), 严格限制前端传入的字数上限)  
* color (VARCHAR(10), 存储前端便利贴的 HEX 颜色代码)  
* create\_time (DATETIME)  
* is\_deleted (TINYINT)

**6\. ugc\_review (通用评价中台表)**

* id (PK, BIGINT)  
* user\_id (BIGINT)  
* target\_type (TINYINT: 1-学校, 2-POI)  
* target\_id (BIGINT, 对应的学校ID或POI ID)  
* score (DECIMAL(2,1), 精度保证如 4.5 分)  
* content (VARCHAR(500))  
* create\_time (DATETIME)  
* *(注意：此处必须建立联合索引 idx\_target (target\_type, target\_id) 以应对高频的前端渲染请求)*

*(注：组队、日记等表结构将在后续 Phase 迭代时按此严格的解耦逻辑与审计规范继续添加)*

## **6\. 开发执行规范 (给开发者的建议)**

1. **接口规范 (RESTful 与统一响应)：** \* 请求路径需清晰表达资源层级关系，例如获取某校区所有 POI: GET /api/v1/campuses/{campusId}/pois。  
   * 所有后端接口必须包裹在统一的响应结构体中（如 Result\<T\> 类），结构需包含：{ "code": 200, "message": "success", "data": {...} }，方便前端统一处理异常和弹窗。  
2. **防踩坑指南与架构取舍：** \* **拒接过度设计：** 作为寻找实习的单人项目，前期绝不要为了炫技去拆分微服务（Spring Cloud 等），采用 **单体架构 (Monolithic Spring Boot)** 是最稳妥的选择。  
   * 采用领域驱动驱动的包结构划分：com.tour.core, com.tour.ugc, com.tour.social，保证即使在单体内，代码层面也是高内聚低耦合的。  
   * 核心数据（如大学列表字典）几乎不修改但查询极高频，后期强烈建议在这层接口加入 Redis 缓存切面（AOP），应对可能出现的流量尖峰。

## **7\. 开发环境与核心技术栈 (Tech Stack)**

* **核心应用框架：** Spring Boot 3.x。作为整个后端的脚手架，实现快速开发与自动配置。（推荐配合 JDK 21 使用，拥抱虚拟线程等新特性）。  
* **持久层框架：** MyBatis (强烈推荐集成 MyBatis-Plus)。作为 Java 与 MySQL 交互的桥梁，极大简化单表 CRUD 操作，同时保留 XML 手写复杂 SQL 的灵活性。  
* **开发语言与运行时：** JDK 21 (采用当前最新的 LTS 版本)。  
* **数据库：** MySQL 8.4 (最新 LTS 数据库版本。支持更强大的 JSON 字段解析及窗口函数)。  
* **集成开发环境 (IDE)：** IntelliJ IDEA (主战工具，内置强大的 Database 面板可替代部分独立数据库软件)。  
* **接口设计与调试：** APIFox (坚持 API First 的正规军规范，用于前期的接口路由设计、文档生成及 Mock 调试自测)。

## **8\. 部署与运维规划 (Deployment & DevOps)**

*本项目最终将脱离本地 localhost 环境，完整部署至云端，实现外网访问，证明开发者具备完整项目的生命周期把控能力。*

* **云服务器提供商：** 腾讯云 (Tencent Cloud)。轻量应用服务器或 CVM 均可。  
* **操作系统环境：** Linux (推荐 Ubuntu 22.04 LTS 或 CentOS)。熟悉生产环境的 Shell 命令操作。  
* **容器化与编排：** Docker。  
  * 将彻底抛弃传统的直接在服务器安装环境的方式。  
  * 后续将针对 Spring Boot 后端项目编写 Dockerfile。  
  * 编写 docker-compose.yml 文件，将 MySQL、Redis 等中间件与后端微服务一起打包，实现一行命令 (docker-compose up \-d) 快速完成生产环境构建与一键部署，极大提升工程化水准。
