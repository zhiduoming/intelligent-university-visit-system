# 小陈 Java后端MVP开发流程（2026-04）

## 协作约定

1. 你主写代码，我负责方案和排障支持。
2. 没有你明确指令，我不改你的代码。
3. 本流程按你当前水平设计，目标是尽快拿到可投递MVP。

## 总目标（建议截止：2026-05-10）

1. 完成可运行的 Java 后端主框架。
2. 跑通核心业务闭环：认证 + 高校/校区/POI + 愿望墙。
3. 完成一次云上部署（腾讯云 + Docker）。
4. 产出可用于投递的项目材料（README、接口说明、项目亮点）。

## 阶段拆分（详细执行版）

### Phase 0：项目定版（2026-04-15）

1. 确定范围只做两所高校（建议：北邮 + 你熟悉的第二所）。
2. 确定技术栈：Spring Boot + MyBatis(+Plus可选) + MySQL + JWT。
3. 在 `docs` 中补充一页《MVP范围说明》。
4. 明确“暂不做”：组队并发、开放日、雷达图、复杂推荐。

完成标准：

1. 你能一句话讲清“第一版做什么、不做什么”。

### Phase 1：后端骨架 + 数据库（2026-04-16 ~ 2026-04-19）

1. 建立分层结构：`controller/service/mapper/entity/dto/vo/common`。
2. 定义统一返回体 `Result<T>`。
3. 接入全局异常处理。
4. 配置文件分环境（`dev`、`prod` 至少占位）。
5. 建 4 张核心表：`user`、`university`、`campus`、`poi`。
6. 插入初始化数据（两所高校、校区、POI）。

完成标准：

1. 应用可启动。
2. 数据库可建表且能查到种子数据。
3. 至少有 1 个健康检查接口可通。

### Phase 2：认证模块（2026-04-20 ~ 2026-04-23）

1. 注册接口（密码加密存储）。
2. 登录接口（签发 JWT）。
3. JWT 拦截校验。
4. 当前用户信息接口（`/me`）。
5. 基础参数校验（用户名/密码长度、空值）。

完成标准：

1. 可完成注册 -> 登录 -> 带 Token 访问受保护接口的闭环。
2. 明文密码不入库。

### Phase 3：核心查询能力（2026-04-24 ~ 2026-04-28）

1. 高校列表接口。
2. 高校详情接口（含校区列表）。
3. 按校区查询 POI 列表接口。
4. POI 详情接口。
5. POI 支持基础筛选（分类）和排序（先做默认/热度）。

完成标准：

1. 前端即使简陋，也能完成“选高校 -> 选校区 -> 看POI”。
2. 接口命名和返回结构统一。

### Phase 4：一个UGC功能（愿望墙）（2026-04-29 ~ 2026-05-03）

1. 建表 `ugc_wish`。
2. 发愿望接口（登录后）。
3. 愿望列表接口（按高校）。
4. 删除愿望接口（逻辑删除，限本人）。
5. 分页能力（最简版 page/size）。

完成标准：

1. 愿望墙功能闭环可跑通。
2. 你能讲清“为什么先做愿望墙，不先做组队”。

### Phase 5：部署与求职包装（2026-05-04 ~ 2026-05-10）

1. 编写 `Dockerfile`。
2. 编写 `docker-compose.yml`（app + mysql）。
3. 腾讯云部署并实现外网访问。
4. 完善 README：项目介绍、技术栈、架构图、启动方式、接口示例、亮点与改进计划。
5. 准备面试讲稿（3分钟版 + 8分钟版）。

完成标准：

1. 给出可访问地址或可复现实操步骤。
2. README 可直接给面试官阅读。

## 每日执行节奏（固定）

1. 90 分钟：主功能编码。
2. 30 分钟：联调 + 自测。
3. 20 分钟：记录今日完成、卡点、明日第一任务。
4. 30 分钟：高频八股（只围绕今天写过的代码）。

## 接口最小清单（先做这些）

1. `POST /api/v1/auth/register`
2. `POST /api/v1/auth/login`
3. `GET /api/v1/users/me`
4. `GET /api/v1/universities`
5. `GET /api/v1/universities/{id}`
6. `GET /api/v1/campuses/{id}/pois`
7. `GET /api/v1/pois/{id}`
8. `POST /api/v1/wishes`
9. `GET /api/v1/universities/{id}/wishes`
10. `DELETE /api/v1/wishes/{id}`

## 防拖延红线（关键）

1. 连续 2 天卡在“改架构/重构”就暂停，先把功能跑通。
2. 没有接口闭环前，不做美化和复杂优化。
3. 一周最多新增 1 个功能点，其他放入 backlog。
